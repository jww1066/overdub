# Design Doc Review — 2026-07-05

Critique of [`design-summary.md`](design-summary.md) and [`prototype-plan.md`](prototype-plan.md),
produced using the `android-skills` plugin (UX/accessibility, coroutines/concurrency, DataStore
persistence, and data-layer error-propagation domains) plus general engineering review judgment.
None of the installed Android skills are audio-specific, so the audio/DSP claims themselves are
evaluated on engineering-methodology grounds, not against packaged domain expertise.

## Critical gaps

**1. Headphone use silently defeats the entire alignment mechanism.** The design's core bet is
"accept mic bleed... [it] enables cross-correlation-based timing alignment... using the beatbox
bleed as a natural reference signal." But musicians instinctively reach for headphones while
recording — precisely to avoid bleed. Neither doc says whether the app *forces* speaker output for
the reference track during recording (overriding a connected Bluetooth/wired headset) to guarantee
bleed exists, or what happens if it doesn't (silent misalignment with no error surfaced?). Given
headphone use already ruled out Bluetooth for latency reasons, this needs an explicit answer, not
an implicit assumption. This is the single biggest hole — it undermines the "no calibration needed"
claim for a large, predictable fraction of real users.

**2. Chain-of-forwarding error compounding is never addressed.** The whole product is "record,
share, overdub, forward further" — multi-hop by design. Both docs analyze alignment for *one*
overdub relationship only. If each hop's GCC-PHAT alignment has some residual error (finite SNR
against phone-speaker bleed), does that error compound over 3-4 hops into audible drift? Test 2 as
scoped (single-device, one recording) can't surface this — it would need a chained multi-hop
simulation. Worth a paragraph in the design doc and a Test 3 candidate.

**3. Shared-file format/codec isn't specified, and it matters a lot here.** Messaging apps commonly
transcode attachments (WhatsApp/Signal often re-encode audio to a lossy codec, sometimes at reduced
sample rate). GCC-PHAT's correlation peak sharpness depends on preserved high-frequency content and
phase — a share-sheet round-trip through a lossy codec is a real, untested variable that sits
directly on top of the SNR-floor question Test 2 is trying to answer. Test 2's "single-device bleed"
setup doesn't include a send/receive round-trip through an actual messaging app, so it validates a
cleaner signal path than production will actually have.

## Concurrency/architecture gaps (kotlin-coroutines skill)

Neither doc says where the GCC-PHAT computation, echo cancellation, or file I/O for the shared
recording will run. Given `CLAUDE.md`'s own main-thread-discipline rule (carried over from the
sibling project), this should be explicit before implementation starts: alignment/AEC is CPU-bound
and belongs on `Dispatchers.Default`, not inline with the Oboe callback or the UI. Relatedly — is
alignment cancellable if the user backs out mid-processing? The skill's scope-ownership guidance
(`suspend fun`, caller owns the scope) is a concrete pattern to bake into whatever
`AlignmentEngine`/`OverdubRepository`-shaped class ends up owning this, so it doesn't need to be
reverse-engineered later the way the noisedroid FFT-synthesis threading fix was.

## Persistence gap (datastore skill)

The lead-in design has session state that must survive recording (key, meter, tempo, whether the
reference chord was enabled) and arguably across app relaunches. Neither doc mentions a persistence
strategy. Given the datastore skill's two traps (a broad `catch` swallowing `CancellationException`;
`CorruptionException` handling for typed stores), this is worth deciding early rather than
defaulting to whatever's fastest to type.

## Accessibility gap (android-ux skill)

The "get ready → recording" cue is described as "explicit visual/audio cue," but not further
specified. For a performer who's watching their instrument/hands (not the screen) at the exact
moment they need the cue, a visual-only signal is the wrong channel — this is exactly the kind of
thing the M3 accessibility audit flags (semantic cues, non-visual channels). Worth explicitly
deciding: does "recording" also get a haptic pulse, not just visual/audio, for users who can't watch
the screen at the critical instant?

## Prototype-plan-specific critique

- **No quantitative pass/fail threshold on either test.** Test 1 says "confirm the measured offset
  is stable" and Test 2 says "map out where the correlation peak degrades" — both are described as
  investigations, not gated tests. Since the plan's own framing is "if either fails, reconsider the
  design direction," it needs a number (e.g., offset variance under ±X samples across 20 reps;
  minimum peak-to-sidelobe ratio) decided *before* running the test, or "pass" will be a judgment
  call made after seeing the data — which is exactly the kind of post-hoc reasoning the rest of
  these docs are careful to avoid elsewhere.
- **No cross-check that the production (Kotlin/NDK) implementation matches the validated Python
  reference.** Test 2's sequencing validates the *algorithm* in Python first, which is good
  practice, but there's no step that ports the same synthetic test vectors into a JVM/instrumented
  test against the real implementation once it exists — without that, a translation bug during the
  Kotlin/C++ port could silently reintroduce the exact risk Test 2 was meant to retire. Worth
  preserving Test 2's synthetic fixtures as regression tests, the way noisedroid's FFT spectral-shape
  tests did for its DSP.
- **Single-device scope is acknowledged but underweighted.** Both tests explicitly defer cross-device
  variance (speaker/mic quality, AGC behavior) as "lower priority." Given Android hardware
  fragmentation is exactly the kind of risk this project's own sibling app (noisedroid) got bitten by
  (OEM battery management, Espresso/API-level quirks), the deferral is reasonable for a first pass
  but should be flagged as a hard gate before wider release, not just an optional nice-to-have.

## What's genuinely solid

The negative-control design in Test 1 (continuous stream vs. two sequentially-scheduled players) is
good experimental hygiene. The explicit "why these two tests and not the others" section correctly
identifies the load-bearing assumptions and prunes scope creep. The UX resolution for pickup notes
(cue-based, not detection-based) is a sound complexity/risk tradeoff. The skills-research section is
honest about its own confidence limits rather than overclaiming.
